# เทมเพลต Prompt สำหรับสอนผู้เรียนสร้าง Skill ให้ "ครอบคลุม"

เอกสารนี้ใช้สอนผู้เรียนให้เขียน prompt สั่ง Claude (ผ่าน skill ชื่อ **skill-creator**) เพื่อสร้าง skill ที่ใช้งานได้จริงและครอบคลุม โดยใช้กรณีศึกษา **"สร้างรายงานยอดขายประจำเดือนจากข้อมูลดิบ ลงเทมเพลตที่เลือกได้ 2 แบบ"** เป็นตัวอย่างหลัก

---

## หลักคิดก่อนเขียน prompt (สอนผู้เรียน 1 นาที)

Skill ที่ดีไม่ใช่แค่ "บอกให้ Claude ทำอะไร" แต่ต้องบอกให้ครบ 6 ด้านนี้ ผู้เรียนจำง่าย ๆ ว่า **"ใคร–เมื่อไร–อะไรเข้า–อะไรออก–ทำยังไง–เช็คยังไง"**

1. **ใคร/บริบท** — skill นี้ช่วยใครทำงานอะไร
2. **เมื่อไร (Trigger)** — ผู้ใช้พิมพ์ประโยคแบบไหนถึงควรเรียก skill นี้ (ใส่ทั้งคำไทยและอังกฤษหลาย ๆ แบบ)
3. **อะไรเข้า (Input)** — ต้องอ่าน/ถามข้อมูลอะไรก่อนเริ่ม
4. **อะไรออก (Output)** — ผลลัพธ์เป็นไฟล์อะไร รูปแบบไหน มีกี่แบบ/กี่เทมเพลต
5. **ทำยังไง (Workflow)** — ขั้นตอน เครื่องมือ/ไลบรารี และเทคนิคเฉพาะที่ต้องใช้
6. **เช็คยังไง (Verify)** — จะรู้ได้อย่างไรว่าผลลัพธ์ถูกต้อง

> เคล็ดลับ: ทุกข้อกำหนดให้ระบุ **"ทำไม"** ด้วย เพราะ Claude จะเข้าใจเจตนาและทำได้ดีกว่าการสั่งห้วน ๆ

---

## เทมเพลต Prompt (ให้ผู้เรียนคัดลอกไปเติมช่องว่าง)

```
ช่วยสร้าง skill ใหม่ให้ฉันด้วย skill-creator โดยมีรายละเอียดดังนี้

ชื่อ skill: <ชื่อสั้น ๆ เป็น kebab-case เช่น monthly-report-generator>

1) วัตถุประสงค์: skill นี้ช่วย <ใคร> ทำ <งานอะไร> เพื่อ <ผลลัพธ์/คุณค่า>

2) ควรทำงาน (trigger) เมื่อผู้ใช้พูดว่า: <ยกตัวอย่างประโยคจริง 5–8 แบบ ทั้งไทย/อังกฤษ
   ทั้งทางการและพิมพ์เร็ว ๆ รวมถึงกรณีที่ไม่ได้เอ่ยชื่อ skill ตรง ๆ>
   และไม่ควร trigger เมื่อ: <งานใกล้เคียงที่ไม่ใช่ 2–3 แบบ>

3) ข้อมูลที่ต้องอ่าน/ถามก่อนเริ่ม: <ไฟล์ข้อมูลดิบ และสิ่งที่ต้องถามผู้ใช้ เช่น เดือน ผู้จัดทำ
   เทมเพลตที่ต้องการ> ถ้าข้อมูลไม่ครบให้ถามก่อน อย่าเดาเอง

4) ผลลัพธ์ที่ต้องการ: <ไฟล์อะไรบ้าง .docx/.pdf> + <มีเทมเพลตให้เลือกกี่แบบ อยู่ที่ไหน>
   ข้อกำหนดรูปแบบ: <ฟอนต์ สี โครงสร้างหัวข้อ จำนวนหน้า ภาษา>

5) ขั้นตอน/เทคนิคที่ต้องใช้: <เครื่องมือ ไลบรารี และข้อกำหนดทางเทคนิคที่ห้ามพลาด พร้อมเหตุผล>

6) การตรวจสอบ: <จะ verify อย่างไร เช่น validate ไฟล์ เรนเดอร์ดูภาพ นับหน้า ตรวจตัวเลขให้ตรงข้อมูลดิบ>

7) ทรัพยากรที่ให้มา: <ไฟล์เทมเพลต ฟอนต์ ตัวอย่างข้อมูล ที่ควร bundle ไว้ในตัว skill>

ขอให้:
- สร้างไฟล์ skill (SKILL.md + scripts + references + assets) ให้ครบ
- เขียน description ให้ trigger ได้แม่นยำ ครอบคลุมคำที่ฉันให้ไว้
- ทดสอบจริงด้วยตัวอย่าง 2–3 เคส แล้วให้ฉันดูผลก่อนปรับแก้
- เมื่อพอใจแล้ว package เป็นไฟล์ .skill ให้ฉันติดตั้ง
```

## ตัวอย่างจริง (เติมเต็มแล้ว) — Skill "สร้างรายงานยอดขายประจำเดือน"

ให้ผู้เรียนอ่านตัวอย่างนี้เป็น "มาตรฐานของคำว่าครอบคลุม" — สังเกตว่ามีการอ้างอิง
**ไฟล์ข้อมูลดิบ**, **เทมเพลตให้เลือก 2 แบบ**, และ **สเปกโครงสร้างรายงานที่ชัดเจน**

```
ช่วยสร้าง skill ใหม่ให้ฉันด้วย skill-creator

ชื่อ skill: monthly-report-generator

1) วัตถุประสงค์: ช่วยพนักงาน/หัวหน้าทีมขาย แปลงข้อมูลดิบรายเดือนให้เป็นรายงานยอดขาย
   ประจำเดือนที่จัดรูปแบบสวยงาม พร้อมส่งผู้บริหารได้ทันที

2) trigger เมื่อผู้ใช้พูดว่า: "ทำรายงานประจำเดือน", "สรุปยอดขายเดือนนี้", "สร้าง monthly report",
   "ทำรายงานยอดขายจากไฟล์นี้", "สรุปผลการดำเนินงานรายเดือนเป็น Word/PDF",
   หรือแนบไฟล์ข้อมูลดิบมาแล้วขอรายงาน
   ไม่ trigger เมื่อ: ขอกราฟ/dashboard แบบโต้ตอบ, ขอวิเคราะห์เชิงลึกอย่างเดียวโดยไม่ทำเอกสาร,
   หรือขอสรุปรายวัน/รายไตรมาส

3) ต้องอ่าน/ถามก่อนเริ่ม:
   - อ่านไฟล์ข้อมูลดิบ เช่น 03_monthly_report_raw_data.txt (ยอดขาย, ลูกค้า, สินค้าขายดี,
     ทีมขาย, ปัญหา, แผนเดือนหน้า)
   - ถามผู้ใช้ว่าจะใช้เทมเพลตแบบไหน: "Formal" หรือ "Modern" (มีให้เลือก 2 แบบ — ดูข้อ 7)
   - ยืนยัน เดือน/ปี, ผู้จัดทำ, ชื่อรายงาน ถ้าไม่มีในไฟล์

4) ผลลัพธ์: ไฟล์ Word (.docx) และ PDF ของรายงาน เลือกได้ 2 เทมเพลต
   โครงสร้างรายงาน (ตามสเปกนี้):
     1. หน้าปก: ชื่อรายงาน, เดือน, วันที่, ผู้จัดทำ
     2. Executive Summary (ครึ่งหน้า): ไฮไลท์ 3–5 ข้อ
     3. ผลการดำเนินงาน: ตารางเปรียบเทียบเป้า vs จริง
     4. สินค้าขายดี Top 4: ตาราง
     5. ทีมขาย: ตารางผลงาน
     6. ปัญหาและแนวทาง: Bullet points
     7. แผนเดือนหน้า: Checklist
   ภาษา: ไทย | ฟอนต์: Anuphan | ความยาว: ไม่เกิน 3 หน้า

5) เทคนิคที่ต้องใช้ (พร้อมเหตุผล):
   - ใช้เทมเพลต HTML ที่ให้มาเป็นฐานดีไซน์ แล้วเติมข้อมูลจากไฟล์ดิบลงไป (อย่าออกแบบใหม่เอง
     เพื่อให้ตรงแบรนด์/มาตรฐานองค์กร)
   - แปลงตัวเลขให้ถูกต้องตามข้อมูลดิบ และคำนวณส่วนที่ต้องสรุป (เช่น % เกินเป้า, MoM) ให้ตรง
   - ใช้ฟอนต์ Anuphan และฝังเข้าไฟล์ผลลัพธ์ (ใน .docx ใช้ OOXML embedded font; ใน PDF กลีฟ
     ถูกฝังอยู่แล้ว) เพราะไฟล์เก็บแค่ "ชื่อฟอนต์" ถ้าปลายทางไม่มีฟอนต์จะกลายเป็นกล่อง (tofu)
     การฝังทำให้ self-contained และ **แสดงผลในหน้า preview ของ Claude Cowork ได้โดยไม่มีปัญหา**
   - ตั้ง proofing language เป็น th-TH ที่ slot w:bidi (เพราะภาษาไทยเป็น complex script ของ Word)
     และปิด spell-check เฉพาะ run ภาษาไทย (ใส่ noProof เฉพาะ run ไทย) เพื่อไม่ให้ขึ้นเส้นแดง
     แต่คำอังกฤษยังถูกตรวจสะกดตามปกติ
   - ถ้าต้องการ PDF ที่ตรงดีไซน์เป๊ะ ให้เรนเดอร์จาก HTML ด้วย headless Chromium

6) ตรวจสอบ: validate ไฟล์ .docx ให้ผ่าน, เรนเดอร์เป็นภาพแล้วดูทุกหน้า, ยืนยันความยาว ≤ 3 หน้า,
   ตรวจว่าตัวเลขในรายงานตรงกับไฟล์ข้อมูลดิบทุกจุด, ภาษาไทยไม่เป็นกล่อง

7) ทรัพยากรที่ให้มา (ให้ bundle ไว้ในตัว skill):
   - เทมเพลต HTML 2 แบบ (รายงาน A4 3 หน้า ออกแบบมาเพื่อพิมพ์เป็น PDF):
     Formal → Sales Report - Formal.html  (หัวข้อแถบ navy ทึบ, ตารางมีเส้นกรอบ + แถวสลับสี,
              ปกมีกรอบเส้นกลางหน้า — ดูเป็นทางการ)
     Modern → Sales Report - Modern.html  (ดีไซน์เดียวกันแต่โทนโปร่งกว่า เหมาะงานนำเสนอ)
     ทั้งคู่ใช้ palette navy #1F3864 และมีโครงสร้าง 6 ส่วนตามด้านบน (KPI + กราฟแท่ง
     เป้า vs จริง + ตารางสินค้า/ทีม + ปัญหา + checklist)
   - ฟอนต์ Anuphan ครบชุด (Anuphan-Regular.ttf, Anuphan-Bold.ttf ฯลฯ) — ให้สลับฟอนต์ของเทมเพลต
     จากค่าเดิม (IBM Plex Sans Thai) มาเป็น Anuphan แล้วฝังลงไฟล์ผลลัพธ์
   - ไฟล์ข้อมูลตัวอย่าง 03_monthly_report_raw_data.txt ไว้ใช้ทดสอบ

ขอให้สร้างไฟล์ skill ให้ครบ, เขียน description ให้ trigger แม่น, ทดสอบกับไฟล์ข้อมูลตัวอย่าง
2–3 เคส (ลองทั้งเทมเพลต Formal และ Modern) ให้ฉันดูผลก่อน แล้ว package เป็น .skill
```

> หมายเหตุสำหรับผู้สอน:
> - ก่อนรัน prompt นี้ ให้เชื่อมต่อ (mount) โฟลเดอร์ที่มีเทมเพลตทั้งสองและไฟล์ข้อมูลดิบ

>   "ไฟล์จริงไม่ตรงกับข้อกำหนด" ให้ผู้เรียนฝึกสั่ง Claude ให้ **สลับมาใช้ Anuphan แล้วฝังฟอนต์**
>   ทุกครั้ง การฝังฟอนต์ทำให้ภาษาไทยแสดงผลถูกต้องทั้งในไฟล์ที่ส่งต่อและในหน้า preview ของ

## ตัวอย่าง Prompt สำหรับ "เรียกใช้" skill (หลังติดตั้งแล้ว)

ส่วนก่อนหน้าคือ prompt สำหรับ **สร้าง** skill — ส่วนนี้คือ prompt ที่ผู้ใช้พิมพ์เพื่อ **เรียกใช้**
skill `monthly-report-generator` ในงานจริง สอนผู้เรียนว่า prompt เรียกใช้ที่ดีควรบอก
**ไฟล์ข้อมูล + เทมเพลต + รูปแบบไฟล์ที่ต้องการ** ให้ครบ จะได้ไม่ต้องถามกลับหลายรอบ

**1) แบบครบถ้วน (แนะนำ)**
```
ทำรายงานยอดขายประจำเดือนจากไฟล์ 03_monthly_report_raw_data.txt
ใช้เทมเพลต Formal, ฟอนต์ Anuphan, ออกเป็น PDF และ Word
ประจำเดือนมีนาคม 2569 จัดทำโดยฝ่ายขายและการตลาด
```

**2) แบบสั้น (skill จะถามข้อมูลที่ขาดเอง)**
```
สรุปยอดขายเดือนนี้เป็นรายงานจากไฟล์ที่แนบ ขอแบบ Modern
```

**3) แบบลำลอง / ไม่เอ่ยชื่อ skill (ทดสอบว่า trigger ติดไหม)**
```
มีไฟล์ข้อมูลยอดขายเดือน มี.ค. อยู่ ช่วยทำเป็นรายงานสวย ๆ ส่งผู้บริหารหน่อย
เอาแบบทางการ ความยาวไม่เกิน 3 หน้า
```

**4) เปรียบเทียบสองเทมเพลต (ให้เลือกทีหลัง)**
```
ทำ monthly report จาก 03_monthly_report_raw_data.txt ให้ดูทั้งแบบ Formal และ Modern
แล้วให้ฉันเลือกว่าจะใช้อันไหน
```

> สอนผู้เรียนสังเกต: prompt ที่ดี "เรียกใช้" จะระบุ **อินพุต (ไฟล์ไหน) + ตัวเลือก (เทมเพลต/ฟอนต์)
> + เอาต์พุต (PDF/Word)** ถ้าไม่ระบุ skill ที่ออกแบบมาดีจะ **ถามกลับก่อนทำ** ไม่ใช่เดาเอง —
> และผลลัพธ์ต้อง **ตัวเลขตรงกับไฟล์ข้อมูลดิบ** เสมอ

## เกณฑ์ให้คะแนนงานผู้เรียน (Rubric สั้น ๆ)

ใช้ตรวจว่า prompt ของผู้เรียน "ครอบคลุม" หรือยัง — ให้ครบ 6 ข้อนี้ถือว่าผ่าน

- [ ] ระบุ trigger เป็นประโยคจริงหลายแบบ (ไม่ใช่คำกว้าง ๆ) และมีกรณี "ไม่ควร trigger"
- [ ] บอกชัดว่าต้องอ่านไฟล์อะไร และต้องถามอะไรก่อนเริ่ม (เช่น เลือกเทมเพลตไหน)
- [ ] กำหนด output เป็นไฟล์ที่จับต้องได้ + โครงสร้างเอกสารชัดเจน + เทมเพลตที่เลือกได้
- [ ] มีข้อกำหนดทางเทคนิค **พร้อมเหตุผล "ทำไม"** (ฟอนต์ ภาษา ความยาว)
- [ ] มีขั้นตอนตรวจสอบผลลัพธ์ รวมถึง "ตรวจตัวเลขให้ตรงข้อมูลดิบ"
- [ ] บอกทรัพยากร/เทมเพลต/ฟอนต์/ไฟล์ตัวอย่างที่ให้มาเพื่อให้ skill ทำงานได้จริง

## ทิป "ยกระดับ" สำหรับผู้เรียนที่อยากได้ skill คุณภาพสูง

- ให้ **ไฟล์จริง** (ข้อมูลดิบ + เทมเพลต + ตัวอย่างผลลัพธ์ที่ดี) แนบไปกับ prompt — Claude เลียนแบบได้แม่นกว่าคำอธิบาย
- เมื่อมีหลายเทมเพลต ให้สั่ง Claude **ถามก่อนว่าจะใช้แบบไหน** และให้ลองทดสอบทั้งสองแบบ
- ขอให้ Claude **ทดสอบและให้ดูผลก่อนปรับ** (วนซ้ำ) แทนการเอาผลรอบเดียว
- ย้ำให้ Claude **ตรวจตัวเลขในรายงานกับข้อมูลดิบ** ทุกครั้ง (รายงานที่เลขผิดคืออันตราย)
- ถ้าเจอข้อจำกัด (ฟอนต์/สิทธิ์/เครื่องมือ) ให้บอก Claude ว่า "หาวิธีอื่นที่ทำได้จริง" แทนการยอมแพ้
